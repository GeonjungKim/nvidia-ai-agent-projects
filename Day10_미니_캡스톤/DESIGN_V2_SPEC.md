## D1. 계약 스키마 v2 — 숙소 앵커와 일자별 앵커의 1급 시민화 (원인 R1)

**변경**
- `TripContract`에 `stay_area: str` 신설 (일본어 역/지명, 예: `駒込`).
  - concierge 추출 규칙: "○○역 쪽에서 묵을 예정" → `stay_area`. `locked[{slot:"stay"}]`로 들어온 것도 `stay_area`로 승격. notes에만 남기는 것 금지.
- `day_anchors: dict[int, str]` 신설 — 계약 확정 시 concierge가 제안하고 사용자에게 보여준다.
  - 기본 규칙: 고정 식당·장소가 있는 날은 그 지역, 마지막 날은 `stay_area` 인접, 나머지는 `areas` 순환.
  - 사이드바 계약 요약에 "Day1: 表参道 / Day2: 駒込" 형태로 표시 (사용자가 대화로 수정 가능).

**하위 반영 (설계 수준)**
- 호텔 검색·추천 앵커 = `stay_area` (현재 merge의 hotel_area·areas[0] 사용 제거).
- @scout 활동 앵커 = `day_anchors`.
- 동선 개념: 하루 경로는 "숙소역 출발 → … → 숙소역 귀가"로 렌더 (첫/끝 레그에 숙소 표시, stay_area 좌표는 역 좌표).

**수용 기준**
- 위 시나리오 재실행 시: 호텔 3곳이 고마고메(또는 인접 야마노테선) 기준으로 추천되고, 라벨에 `駒込` 표기.
- 사이드바에 stay_area와 day_anchors가 계약 확정 전에 보인다.

## D2. 공간 모델 — 앵커 해석기 + 클러스터 강제 (원인 R2)

**변경**
- **앵커 해석기(deterministic, LLM 아님)**: 앵커 문자열 → 검색 키 결정.
  1) area2 정확/LIKE 매치 시 `area2` 키 사용 → 2) 실패 시 `station` LIKE 사용 (실측상 동네 단위는 대부분 이 경로) → 3) 둘 다 실패 시 인접 역 후보를 station_coords에서 부분일치로 제시.
  - 해석 결과(키 종류·건수)를 파이프라인 로그에 남긴다: `[앵커] 表参道 → station 검색 282곳`.
- **@foodie 검색 계약**: task에 앵커별 "검증된 검색 키"를 명시해 주입한다 (foodie가 스스로 area2를 지어내지 않게). 프롬프트 지시가 아니라 **task 데이터로 제공**.
- **클러스터 이탈 게이트(코드)**: 최종 픽의 최기역이 해당 day_anchor 역에서 직선 4km 초과면 —
  a) 같은 클러스터의 대안 후보로 교체 시도 → b) 불가하면 카드에 `⚠️ 앵커에서 벗어남 (전철 ~N분)` 라벨을 강제 표기. 조용한 이탈 금지.
- external 장소(고정 포함) 좌표 = 해당 앵커 역 좌표로 근사 → 동선·지도에 포함 (현재는 동선에서 누락됨).

**수용 기준**
- 오모테산도 앵커의 Day1 식사·카페가 전부 表参道/明治神宮前/原宿/青山 반경 내이거나, 이탈 라벨이 붙는다.
- 로그에 앵커 해석 라인이 찍힌다. external 고정 식당이 Day1 동선의 출발점으로 들어간다.

## D3. 시간 모델 — 슬롯 규칙·체류/이동시간·종일형 POI (원인 R3)

**변경**
- 활동 슬롯에 타입 부여: `morning(10:00)` / `late_afternoon(16:30)` / `evening(20:00)`.
- **슬롯-카테고리 적합성 규칙표 (코드 상수)**:
  | 슬롯 | 적합 | 금지 |
  |---|---|---|
  | morning | 신사·정원·시장·미술관 (혼잡 회피 팁 포함) | 야경·전망대(비추 라벨) |
  | late_afternoon | 쇼핑거리·미술관·전망대(일몰 시작이면 최적) | — |
  | evening | **전망대·야경·번화가·이자카야 거리** (예: 모리타워 전망대는 황혼 후 도쿄타워 뷰가 베스트 — 이런 팁을 why에 요구) | **공원·정원·신사** (입장마감 리스크) |
- **종일형 POI 차단 목록 (코드 상수)**: 디즈니랜드/디즈니씨/USJ/후지큐/워너 스튜디오 투어(사전예약제) 등 — 슬롯 배정 금지. @scout이 가져오면 픽에서 제외하고, 대신 일정 하단 "🎢 하루를 통째로 쓰는 옵션" 섹션에 별도 노출 (예약 필수·오픈런 팁과 함께). 계약에 `theme_park: bool` 의향 필드 추가 — true면 그날 전체를 테마파크 일정으로 편성(식사는 파크 내/역 앞으로 단순화).
- **@scout 출력 스키마 확장**: `{title, url, why, area, open_hours?, last_entry?, dwell_min, best_time, tip}`.
  - `why`·`open_hours`는 **검색 스니펫에서 확인된 것만**, 불확실하면 `"확인 필요"` — 창작 금지 (D4에서 검증).
  - `tip` 1줄 필수 (예: "저녁 입장 마감 18:30 — 낮 슬롯에만", "오픈런 아니면 대기 2h+").
- **이동시간 표기**: 활동↔식사 레그에도 역 좌표 기반 추정 적용 — `도보 ~N분` / 2.5km 초과 시 `전철 ~N분 (환승 확인)`. 렌더의 타임라인 항목 사이에 이동 힌트 삽입.

**수용 기준**
- evening 슬롯에 공원/정원 배정 0건. 종일형 POI가 슬롯에 배정 0건 (별도 섹션에만).
- 각 활동 카드에 tip 1줄. 신주쿠 교엔이 나오면 반드시 morning/late_afternoon + 입장마감 팁.

## D4. 검증 비대칭 해소 — @critic 범위 확장 + 코드 dedup (원인 R4)

**변경**
- **코드 게이트 (critic 이전)**: ① 활동 전 일정 dedup (동일 title 2일 배정 금지) ② 슬롯-카테고리 규칙 위반 제거 ③ 종일형 차단.
- **@critic 검증 확장**: 식당 검증에 더해 활동도 판정한다. @scout이 근거로 쓴 **스니펫 원문을 evidence로 첨부**해 critic에 전달 (critic은 읽기 전용 유지 — 새 도구 불필요).
  - 판정 항목: why/open_hours가 evidence에 실재하는가(창작 탐지) · 슬롯 시간대 규칙 위반 · 앵커 정합("교통 편리" 주장과 area 일치).
- **comment(선정 이유) 생성 주체 변경**: 병합 LLM의 자유 서술 금지 → 코드 템플릿("고정 1곳 · 장르 {…} 순환 · Day1 {앵커} 중심")로 생성하거나, LLM 서술을 쓰려면 critic 검수 대상에 포함. "시스템 내 식당으로 정정" 같은 허위 서술 재발 방지.

**수용 기준**
- critic 리포트가 "식당 N곳 + 활동 M건 판정"을 모두 포함. 활동 중복 0건. 허위 comment 0건.

## D5. 식당 품질 하한 게이트 (원인 R5)

**변경 (실측 기반 기본값)**
- 검색·백필·병합 폴백 공통 하한: `tabelog_review_count ≥ 50 AND bayes_score ≥ 3.4` (재고 11,484곳 — 충분).
- 지역이 희소해 하한 미달만 남으면: 1단계 완화(`≥20 / ≥3.3`)를 **라벨과 함께** 적용 (`⚠️ 데이터 희소 지역 — 기준 완화`). 조용한 완화 금지.
- 저녁 슬롯 추가 규칙: 테이크아웃/체인 위주 업태 제외 휴리스틱 — 장르가 {弁当, 惣菜, デリカテッセン, コロッケ} 단독이거나 예산 `～￥999`이면서 좌석형 식사 장르가 없으면 저녁 후보 배제 (구현 시 장르 목록은 DB `restaurant_genres` 실빈도 보고 확정).
- 장르 다양성 게이트(기존)와 충돌 시 우선순위: **품질 하한 > 장르 다양성**. 다양성 대안이 하한 미달이면 중복을 허용하고 사유 라벨.

**수용 기준**
- 위 시나리오 재실행 시 저녁 픽 리뷰 수 ≥ 50 (또는 희소 완화 라벨 표기). ★3.0(리뷰 한 자리) 픽 0건.

## D6. 고정 장소 resolve v2 (원인 R6)

**변경 — 다단 매칭 + 사용자 확인 루프**
1) 사용자 원문 keyword → 2) concierge의 일어 표기 keyword → 3) 토큰 분해 부분일치 (예: "ナナイド" 단독) → 4) **앵커 역 + 장르 검색 후 이름 유사도 상위 3곳**.
- 4)까지 가면 자동 확정하지 않는다 — **채팅으로 후보 3곳을 제시하고 사용자 확인**을 받는다 ("혹시 이 중 하나인가요? 아니면 지정하신 이름 그대로(DB 외) 반영할게요"). 계약 확정 단계(생성 버튼 이전)에서 수행해 파이프라인 중단을 피한다.
- external 확정 시: 카드에 `DB 미등록 — 지정 이름 그대로 반영` 명시 고지 + 앵커 역 좌표 근사로 동선 포함 (D2와 연동).

**수용 기준**
- 매칭 실패 시 사용자 확인 질문이 채팅에 나타난다. external 카드에 고지 문구. 관련 comment 허위 서술 0건.

## D7. 항공 정보 v2 — 메타서치 제품 패턴 적용 (원인 R7)

**벤치마킹 분석**
| 제품 | 패턴 | 우리가 가져올 것 |
|---|---|---|
| 스카이스캐너/카약 | 캐시 가격 메타서치 · **유연 날짜(한 달) 가격 캘린더** · **가격 알림** | 한 달 뷰 딥링크(`/YYMM/YYMM/` 월 단위 URL), 가격 알림 설정 안내 |
| 캐치프로그 | 노선별 역대가 DB → **"지금 사도 되는 가격" 기준선 판단** + 특가 알림 | 기준선 코멘트: 시세 범위 대비 "○○만 원대면 즉시 예약 권장" |
| 트립닷컴/네이버 항공권 | OTA 직판/국내 친화 비교 | 비교 링크 추가 (핸드오프 다변화) |

**변경**
- "₩263,100부터" 같은 **단일 '부터' 가격 금지**. @scout 항공 조사를 2회 검색(서로 다른 소스)으로 늘려 **범위**로 제시: "8월 주말 인천–도쿄 왕복 대략 25~45만 (성수기)".
- **판단 기준선 1줄** 필수: "이 시기 20만 초반이면 평년 대비 저렴 — 즉시 예약 권장" (검색 근거 기반, evidence 첨부 → critic 검증 대상).
- 링크 확장: 기존 특정일 딥링크 + ① 스카이스캐너 **월 전체 가격 캘린더** 딥링크 ② 가격 알림 안내 문구 ③ 네이버 항공권·트립닷컴·마이리얼트립 검색 링크 ④ 공항 선택 팁 (김포–하네다 = 도심 접근 우위 / 인천–나리타 = 가격 우위 경향) — 링크 생성은 `links.py` 결정론.

**핸드오프 4단계 스펙트럼 — 무엇에 계약이 필요한가 (2026-07-20 문답 반영, 종전 서술 정정)**

| 단계 | 내용 | 필요한 것 | 우리 포지션 |
|---|---|---|---|
| ① 링크 아웃 | 날짜·구간 담은 딥링크로 예약처(트립닷컴·마이리얼트립 등) 이동 — 캐치프로그류의 기본 동작 | **없음** — 보편적 관행, 누구나 URL 조립 가능 | ✅ 현재 방식 |
| ② 제휴 링크 | ①에 트래킹 파라미터 → 성사 시 커미션 수익화 | 셀프서브 제휴 가입 (Trip.com 어필리에이트, Travelpayouts, 국내 제휴 네트워크 등 — '기업 계약'이 아니라 승인형 가입, 개인도 가능) | 로드맵 — 발표의 수익모델 언급용 |
| ③ 가격 데이터 표시 | 우리 화면에 실시세·최저가를 직접 렌더 | 가격 API — **Amadeus Self-Service(무료 쿼터, 개인 가입 가능)**, Travelpayouts Data API 등 | 로드맵 P2+ (금일 스코프 밖이나 실현 가능) |
| ④ 예약 실행(발권) | 앱 안에서 결제·발권 완료 | IATA 인가/여행사 제휴(콘솔리데이터)·PCI 등 — 이것이 진짜 "파트너 계약" | 범위 외 유지 |

- 정정: 종전의 "실예약 API는 파트너 계약 필요"는 ④ 기준의 서술. ①은 계약 불필요(현재 방식이며 캐치프로그류와 동일 포지션), ③도 Amadeus Self-Service 티어면 개인 접근이 가능하다. D7의 목표는 ① 안에서 "범위 시세 + 판단 기준선"으로 캐치프로그의 **판단 보조 가치**까지 따라가는 것.

**수용 기준**
- 항공 섹션 = 범위 시세(2소스) + 기준선 1줄 + 링크 4종(특정일/월 캘린더/비교 2). "부터" 단일가 0건.

## D8. 전체 랭킹 리스트 뷰 — 기존 타베로그 앱 로직 이식 (원인 R8)

**변경**
- UI에 **"📃 지역 랭킹 탐색" 탭(또는 일정 하단 확장 패널)** 신설 — 일정과 독립적으로 열람.
  - 프리셋: 계약의 pref · day_anchors · `genres_pref`가 **미리 선택된 상태**로 진입 (사용자 요구: 선호 카테고리 사전 선택 제공).
  - 컨트롤: 앵커(역) 선택 · 장르 멀티선택(해당 지역 상위 장르, 기존 M3의 동적 로드 방식) · 예산 · 정렬(bayes 기본 — 기존 앱과 동일).
  - 카드: 기존 Tabelog_Recommendation M3 카드 필드(이름→타베로그 링크, 장르, ★평점(리뷰수), bayes, 예산, 역) + 구글지도 링크. 상위 30.
  - 구현 힌트: `tabelog_server.search_lib`가 이미 동일 랭킹 로직 — UI만 붙이면 된다 (LLM 무관, 즉시 응답).
- 각 식사 카드에 **"다른 후보 보기"**: 그 슬롯의 나머지 후보 2~3곳 + "이 지역 전체 랭킹 →" 링크(위 탭으로 이동, 필터 프리셋 적용).

**수용 기준**
- 일정 생성 없이도 탭에서 지역×장르 랭킹 열람 가능. 카테고리 변경 시 즉시 갱신(LLM 호출 0회). 슬롯 카드에서 대안 후보 확인 가능.

## D9. 현 단계 규정 (질문에 대한 답)

- `run_demo.py` = CLI 리허설/테스트 스크립트, `ui.py` = 본편. **둘은 같은 `run_pipeline`을 공유**하며 별도 "데모 전용 로직"은 없다.
- 완성도 규정: 현재 = **MVP(알파)** — 식당 축은 3중 방어로 견고하나, 공간(R1·R2)·시간(R3)·활동 검증(R4) 축이 미설계였다. 본 스펙(D1~D8) 반영 시 베타로 규정한다.
