# 🍜 TabeTabi (食べ旅) — 멀티에이전트 미식 여행 컨시어지

> **Day10 미니 캡스톤** · 타베로그 고신뢰 리뷰어 실데이터(유니크 식당 **45,725곳**) 위에서,
> 역할이 나뉜 에이전트 팀이 MCP 도구를 들고 협업해 "맛집 중심 여행 일정"을 만들어 주는 대화형 서비스.

- 사용자가 **고정한 식당·일정은 절대 변경하지 않고**, 나머지 슬롯만 추천한다.
- 모든 추천 식당은 **타베로그 링크 + 구글지도 링크** 제공, 하루 **동선 경로**(경유지 지도 URL)와 지도 표시.
- 항공/숙소는 예약 API 대신 **날짜·구간이 반영된 검색 딥링크**(Google Flights·스카이스캐너·Booking.com).
- 주변 활동·날씨는 웹검색(Tavily)으로 보강.

## 실행

```bash
# (선택) pip install -r requirements.txt
# (선택) .env에 TAVILY_API_KEY 추가 — ENV_SETUP.md 참고. NVIDIA 키는 ../API.txt에서 자동 로드.
streamlit run ui.py        # 채팅 UI (http://localhost:8501)
python run_demo.py         # CLI 데모 (UI 없이 전체 파이프라인)
```

## 아키텍처 — Day9의 "계약 우선 → 병렬 → 게이트"를 런타임에 적용

```

        │  @concierge가 대화에서 계약 추출 (도구 없는 LLM — 계획 단계엔 tools 금지, Day7)
        ▼
┌─ SHARED CONTRACT 고정 ──────────────────────────────┐
│ 고정(locked) 식당은 '코드'가 DB 조회로 확정 — LLM 불개입 │
└──────────────┬──────────────────────────────────────┘
               │  배치 1 — asyncio.gather 병렬
     ┌─────────┴─────────┐
     ▼                   ▼
  @foodie             @scout
  Tabelog DB MCP      웹검색 MCP
  (열린 슬롯 후보 2개씩)  (주변 활동·날씨)
     └─────────┬─────────┘
               ▼  배치 2 — 의존 단계
  병합(@concierge, 도구 없는 LLM) → 후보 밖 선택·EXTERNAL·중복은 코드가 교정, 누락은 결정론 백필
               ▼
  @critic 게이트 (읽기 전용, get_restaurant만) — 불합격 시 병합 1회 재시도(self-correction)
               ▼
  로지스틱스 (결정론): 역좌표 → NN 동선 정렬 → 경유지 지도 URL · 항공/숙소 딥링크
               ▼
  일정표 카드 + st.map + 에이전트 실행 로그
```

## 에이전트 팀 — 허용 도구 목록이 곧 권한 경계 (Day9)

| 에이전트 | 역할 | 허용 도구 | 왜 이 경계인가 |
|---|---|---|---|
| @concierge | 대화·계약·병합 지휘 | (없음 — 순수 LLM) | 판단만 한다. 계획 단계에 tools를 주면 실행해버린다(Day7 Lab2.5) |
| @foodie | 식당 후보 발굴 | `search_restaurants` `list_areas` `list_genres` | DB 결과 밖 식당을 "말할 수 없다" → 환각 구조 차단 |
| @scout | 주변 활동·날씨 | `web_search` | DB 접근 불가 — 식당 추천에 관여 불가 |
| @critic | 검증 게이트 | `get_restaurant` 단 1개 | 읽기 전용. 수정 능력 자체가 없다 |

## 환각 3중 방어 (발표 핵심 포인트)

1. **권한 경계** — @foodie는 DB 도구 결과에서만 후보를 뽑는다.
2. **코드 재조인** — 화면에 보이는 이름·평점·링크는 전부 `restaurant_id`로 DB에서 다시 조회한다. LLM이 전사한 텍스트를 신뢰하지 않는다.
3. **@critic 게이트 + 코드 교정** — 유령 id·계약 위반을 검증하고, 후보 밖 선택은 코드가 1순위로 교정, 누락 슬롯은 베이지안 상위로 결정론 백필.

## 판단 3문 (왜 멀티에이전트인가)

① 병렬 분해 가능한가 — **예**: 계약 고정 후 식당 조사와 활동/날씨 조사는 독립 (실측: 병렬 로그 교차 출력)
② 가치가 토큰 비용보다 큰가 — **예**: 실데이터 45,725곳 기반 실사용 가능 서비스 (1회 생성 ≈ 1분, LLM 호출 10여 회 + 웹검색 ≤6회)
③ 사람이 검수 가능한 단위인가 — **예**: 일정표 슬롯 단위 + @critic 판정 + 실행 로그 공개

## 기존 자산 재사용 (Tabelog_Recommendation)

- `app.db` — 리뷰어 3명 병합 47,703 레코드 → 유니크 45,725곳 `restaurants_agg` (읽기 전용 URI로만 접근)
- `app/maplinks.py` — 구글/애플 지도·경유지 경로 URL (pytest 검증본을 **import로 재사용**)
- `data/station_coords.json` — 역 1,142곳 좌표 (동선 정렬·지도 표시)
- C-2 준수: 리뷰어 내부 필드(reviewers_json 등)는 MCP 서버가 아예 반환하지 않는다

## 🚀 최신 업데이트: 장기 여행(최대 30일) 지원 및 검색 최적화

최근 패치를 통해 최대 **30일**에 이르는 장기 미식 여행 일정도 끄떡없이 생성할 수 있도록 시스템이 대폭 강화되었습니다.
1. **제한 해제 및 안정화**: 기존 14일이었던 여행 기간 제한을 30일로 확장하고, 에이전트들의 도구 사용 한도(`max_steps=100`)와 컨텍스트 토큰(`max_tokens=8000`)을 대폭 상향하여 거대한 일정도 잘림 없이 완성합니다.
2. **중복 검색 방지 (컨텍스트 다이어트)**: 며칠간 동일한 지역(예: 신주쿠 연박)에 머무를 때 앵커 검색 키를 하나로 그룹화하여 **중복 도구 호출을 원천 차단**했습니다. LLM의 컨텍스트 낭비를 막고, API 타임아웃 오류를 근본적으로 해결했습니다.
3. **타임아웃 상향 및 이중화**: 대규모 일정 생성 시의 물리적 추론 시간을 고려해 NIM API 타임아웃을 600초로 상향하고, 장애 시 폴백 모델(`FALLBACK_BASE_URL`)로 즉시 우회하여 복원력을 높였습니다.
